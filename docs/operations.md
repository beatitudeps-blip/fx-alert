# operations.md
## FX半システム運用 運用仕様書

---

## 1. 目的
本ファイルは、FX半システム運用における実運用ルールを定義する。

対象は以下。
- GitHub Actions の実行タイミング
- UTC / JST / DST の扱い
- Twelve Data の取得方針
- LINE Messaging API の通知方針
- 手動確認項目
- 障害時の挙動
- ログ・再試行・アラートの扱い
- NotebookLM / Google Sheets への連携方針

本ファイルは `strategy.md` と `data_spec.md` を日々の運用に落とし込むための運用正本である。

---

## 2. 基本方針

### 2.1 運用の前提
本システムは、以下を前提とする。

- 判定は自動
- 発注は当面手動
- 執行口座はみんなのFX
- 市場データ判定は Twelve Data 等の外部データソース
- 実績正本はみんなのFX約定履歴CSV
- 見送りも成果物として保存する

### 2.2 優先順位
運用上の優先順位は以下。

1. 誤判定を減らす
2. 無通知・無記録を防ぐ
3. 判定と実績を紐付ける
4. レポートを自動化する
5. 改善しやすい形でログを残す

---

## 3. 時刻管理方針

### 3.1 基本原則
- 内部時刻はすべて UTC で保持する
- ユーザー向け表示は JST に変換する
- バックテストとライブ判定は同じ日足定義を使う
- DST込みで固定時刻を決め打ちしない
- 日足確定の正本は市場データ提供元の足とする

### 3.2 UTC と JST の扱い
- CSV正本の日時列は UTC を使う
- Markdown レポートでは JST 表示を使ってよい
- 同じイベントに UTC と JST を両方保存する場合は、UTC を正本とする

### 3.3 DST の扱い
- 実行時刻の判定に「JSTの毎朝○時固定」だけを前提にしない
- 日足更新が済んでいるかを実行時に確認する
- 夏時間 / 冬時間でデータ提供元の更新時刻が変わっても、最新足更新確認で吸収する

### 3.4 通知時刻と DST

- GitHub Actions の cron は UTC 固定で起動する
- Forex 日足クローズは 17:00 ET（標準時 = UTC 22:00 / 夏時間 = UTC 21:00）
- cron を UTC 22:30 に設定しているため、夏時間中は日足確定から約1.5時間後、冬時間中は約30分後に実行される
- この通知時刻のズレは仕様として許容する
- 理由: 本システムは「毎日同じ日本時間に通知する」ことよりも「未確定足を避けて、日足更新後に正しく判定・通知する」ことを優先する
- bar_checker による最新日足更新確認がこのズレを吸収する

## 4. GitHub Actions 実行方針

### 4.1 基本方針
- 日次判定は 1日1回
- GitHub Actions は UTC cron で起動する
- ただし、実際の判定実行可否は「最新日足更新確認」で決める

### 4.2 重要原則
- cron が走った = 判定実行、ではない
- cron は「起動トリガー」
- 判定処理は「最新日足が更新済み」のときだけ行う

### 4.3 推奨ワークフロー
以下のように分離する。

- `daily_signal.yml`
  - 日次判定
  - シグナル生成
  - LINE通知
  - `signals.csv`
  - `daily_signal_report_YYYYMMDD.md`

- `import_broker_csv.yml`
  - みんなのFX CSV取込
  - `raw_fills.csv` 更新
  - `trades_summary.csv` 更新

- `weekly_review.yml`
  - 週次レビュー生成

- `monthly_review.yml`
  - 月次レビュー生成

- `backtest.yml`
  - バックテスト専用
  - ライブ運用とは分離

### 4.4 concurrency
同一workflowの重複起動を防ぐため、GitHub Actions では `concurrency` を設定する。

方針:
- 日次判定 workflow は同時実行禁止
- import workflow も同時実行禁止
- backtest workflow はライブ系と分離

### 4.5 実行頻度
初期版は以下を推奨する。

- 日次判定: 1日1回
- 週次レビュー: 毎週1回
- 月次レビュー: 毎月1回
- CSV取込: 手動実行または必要時実行

## 5. 最新日足更新確認ルール

### 5.1 目的
未確定足や前日足未更新状態で判定しないため。

### 5.2 ルール
判定前に以下を確認する。

- 取得した日足データの最新バー時刻
- 前回判定済みバー時刻
- 最新バーが新しいか

### 5.3 判定
- 最新日足が更新済み → 判定実行
- 最新日足が未更新 → 判定しない
- データ取得失敗 → `NO_DATA` または `ERROR` として記録

### 5.4 保存
以下を保存すること。
- source_data_timestamp_utc
- run_id
- 判定実行有無
- スキップ理由

## 6. Twelve Data 運用方針

### 6.1 基本方針
- 判定の正本は Twelve Data 等の市場データ
- バックテストでは API を乱用せず、CSVベースで行う
- 必要データはまとめ取得する
- 同じデータを短時間に繰り返し取得しない

### 6.2 データ取得対象
初期版で取得する主な対象は以下。
- 日足OHLC
- 週足OHLC
- Daily EMA20
- Weekly EMA20
- Daily ATR14

### 6.3 取得失敗時
取得失敗時は再試行を行う。

推奨:
- 最大3回まで再試行
- 短時間連打を避ける
- 指数バックオフを使う

### 6.4 キャッシュ方針
- 同一実行内では再取得を避ける
- 必要なら一時キャッシュを使う
- バックテスト用データは別保存して再利用する

### 6.5 APIキー管理
- APIキーは GitHub Secrets / 環境変数で管理する
- コードや Markdown に直書きしない

想定環境変数:
- `TWELVE_DATA_API_KEY`

## 7. LINE Messaging API 通知方針

### 7.1 基本方針
通知は LINE Messaging API を利用する。

### 7.2 通知対象
初期版では以下を通知対象とする。

- 日次シグナル結果
- `ENTRY_OK` がある場合の強通知
- 重要な `ERROR` / `FATAL`
- 必要に応じて集計結果

### 7.3 通知ルール
- `ENTRY_OK` がある場合は、エントリー候補価格を含めて通知する
- `SKIP` のみの場合は、一覧サマリー通知でよい
- `NO_DATA` / `ERROR` は障害通知として扱う
- 複数通貨がある場合は、1通貨1メッセージ連打よりサマリー通知を優先する

### 7.4 通知失敗時
- 失敗内容を `error_log.csv` に記録する
- 再送の有無は実装で制御する
- 初期版では、通知失敗 = エラー記録 + workflow失敗扱いでもよい

### 7.5 秘密情報管理
想定環境変数:
- `LINE_CHANNEL_ACCESS_TOKEN`
- `LINE_CHANNEL_SECRET`
- `LINE_USER_ID` または `LINE_TO`

これらは GitHub Secrets で管理すること。

## 8. 手動確認項目

### 8.1 基本方針
初期版では、すべてを自動判定しない。
裁量が必要な項目は明示的に `manual_check` として残す。

### 8.2 初期版の manual_check 対象
- 経済指標リスク
- 特殊な地政学リスク
- 急変相場の違和感
- 実運用時の最終発注可否

### 8.3 経済指標の対象例
- FOMC
- 米CPI
- 米雇用統計
- 日銀政策決定会合

### 8.4 運用ルール
- 自動出力には `event_risk = manual_check` を出す
- ユーザーが最終確認する
- 見送り時は理由コード `E` を付与する
- 実際に確認したかどうかは `trades_summary.csv` の `event_risk_checked` に残す

---

## 9. みんなのFX CSV 取込運用

### 9.1 基本方針
みんなのFX 約定履歴CSVは実績の正本とする。

### 9.2 取込タイミング
初期版では以下のいずれかを許可する。
- 手動アップロード後に取込
- ローカル実行で取込
- GitHub Actions 手動実行で取込

### 9.3 取込時の処理
- CSVの列を標準化
- `raw_fills.csv` に追記 / 更新
- `matched_signal_id` を可能な範囲で付与
- `trade_id` を生成
- `trades_summary.csv` を更新

### 9.4 重複取込対策
- `fill_id` を一意キーとして重複判定する
- 同一CSVの再取込時に二重計上しない
- 重複時は `import_status = DUPLICATE` としてもよい

---

## 10. 障害時運用

### 10.1 重要原則
- 無通知 = シグナルなし、と扱ってはいけない
- 障害は障害として記録する
- 少なくとも `error_log.csv` に残す

### 10.2 障害分類
以下を区別して記録する。

- 正常終了
- データ取得失敗
- 判定失敗
- 通知失敗
- CSV取込失敗
- 集計失敗
- レポート生成失敗

### 10.3 severity
推奨値:
- `INFO`
- `WARN`
- `ERROR`
- `FATAL`

### 10.4 stage
推奨値:
- `DATA_FETCH`
- `SIGNAL_BUILD`
- `NOTIFY`
- `IMPORT`
- `AGGREGATE`
- `REPORT`
- `UNKNOWN`

### 10.5 挙動
- `WARN`: 記録のみ
- `ERROR`: 記録 + workflow失敗候補
- `FATAL`: 即時失敗扱い

初期版では、以下を推奨する。
- データ取得失敗 → `ERROR`
- シグナル生成失敗 → `FATAL`
- 通知失敗 → `ERROR`
- CSV取込失敗 → `ERROR`
- 月次レポート生成失敗 → `ERROR`

## 11. 再試行方針

### 11.1 再試行対象
再試行してよいもの:
- 市場データ取得失敗
- 一時的な通知API失敗

再試行しないもの:
- 戦略ロジックの例外
- CSVパースの構造エラー
- 必須環境変数不足

### 11.2 再試行回数
初期版推奨:
- 最大3回

### 11.3 再試行間隔
- 固定短時間待機でもよい
- 可能なら指数バックオフを使う

---

## 12. ログ方針

### 12.1 ログの目的
- 失敗原因の追跡
- 無通知・無記録防止
- 週次 / 月次の振り返り材料
- Claude Code による自己診断材料

### 12.2 残すべきログ
- workflow 実行時刻
- run_id
- strategy_version
- 取得対象通貨
- データソース時刻
- 判定結果
- 理由コード
- 通知成否
- CSV取込成否
- 例外内容

### 12.3 出力先
- `signals.csv`
- `error_log.csv`
- 日次 / 週次 / 月次 Markdown
- GitHub Actions の実行ログ

---

## 13. NotebookLM 連携方針

### 13.1 基本方針
NotebookLM は生データ置き場にしない。

### 13.2 入れるもの
- `weekly_review_YYYYMMDD.md`
- `monthly_review_YYYYMM.md`
- 戦略仕様書
- 変更履歴
- 勝ち / 負けパターンの要約

### 13.3 入れないもの
- `signals.csv`
- `raw_fills.csv`
- `trades_summary.csv`
- `error_log.csv`
- 生の約定履歴CSV

### 13.4 運用
- 週次レビューだけ投入してもよい
- 月次レビューは優先的に投入する
- NotebookLM は知識化された要約置き場として使う

---

## 14. Google Sheets 連携方針

### 14.1 基本方針
Google Sheets は可視化・確認・月次レビュー補助に使う。

### 14.2 連携対象
- `signals`
- `raw_fills`
- `trades_summary`
- `dashboard`

### 14.3 dashboard の役割
- 月間損益
- 累積損益
- 総R
- 勝率
- PF
- 最大DD
- 最大連敗
- 通貨別成績
- 理由コード別件数

### 14.4 運用ルール
- Sheets は分析・閲覧用
- 正本は CSV / ローカル生成物
- 手修正した場合は正本へ反映しない限り、正式値にしない

---

## 15. 手動発注運用ルール

### 15.1 基本方針
初期版では、発注はユーザーがみんなのFXで手動実行する。

### 15.2 ユーザーが確認するもの
- 通貨ペア
- 売買方向
- エントリー価格
- SL
- TP1
- TP2
- ロット
- event_risk
- チャート違和感

### 15.3 実運用の注意
- シグナルが出ても必ず発注しなければならないわけではない
- 見送った場合は、可能なら理由を記録する
- ロット変更や価格変更などの手動介入は、後で `trades_summary.csv` に痕跡を残せるようにする

## 16. フェーズ別運用の違い

### 16.1 フェーズ1
- 日次シグナル生成
- 手動発注
- CSV手動取込
- 週次 / 月次レポート生成
- event_risk は manual_check

### 16.2 フェーズ2
- CSV取込の効率化
- Google Sheets の自動反映
- エラー通知強化
- 手動介入ログの整備

### 16.3 フェーズ3
- 指標イベント自動連携
- 相関チェック高度化
- 週足支持抵抗の高度化
- より厳密な MFE / MAE 管理

## 17. 運用開始前チェックリスト

### 17.1 Secrets
- `TWELVE_DATA_API_KEY` が設定済み
- `LINE_CHANNEL_ACCESS_TOKEN` が設定済み
- `LINE_CHANNEL_SECRET` が設定済み
- `LINE_USER_ID` または `LINE_TO` が設定済み

### 17.2 戦略仕様
- `strategy.md` が最新
- `data_spec.md` が最新
- `strategy_version` が固定されている

### 17.3 ワークフロー
- `daily_signal.yml` が起動する
- `concurrency` が設定されている
- `error_log.csv` が出力される
- 日足更新確認が入っている

### 17.4 実績管理
- `signals.csv` が出力される
- `raw_fills.csv` が更新できる
- `trades_summary.csv` が集約できる
- 月次レビューが生成できる
